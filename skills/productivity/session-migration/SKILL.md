# 会话迁移

当用户要求迁移会话、打包当前状态、或需要换新会话继续时使用。

## 通用迁移 Prompt

直接复制以下内容，粘贴给当前对话的AI：

```
请帮我整理当前对话的完整状态，我要迁移到新会话继续。

先判断当前阶段：发散探索 / 收敛执行 / 校验纠错。如果同时有多条任务线，分别标注每条所处的阶段。
然后按以下结构输出，只保留确定信息，不猜测不脑补：

## 目标
主要在做什么，要达成什么。如果有多条任务线，分别列出，标出主线

## 已确认的结论
确定了的决策、方向、方案、判断。每条独立写清

## 已否决的方案（附原因）
讨论过但放弃的方向，以及迭代中不认可的角度/写法/方向。写清楚为什么——这特别重要，防止新会话重复踩坑

## 进度
按任务或文件分别标注状态：
- ✅ 已完成的：简要写结果
- 🔧 进行中的：做到哪了
- ⏳ 待办的：还没开始
涉及多个文件时，列出每个文件各自状态

## 关键约束
背景条件、偏好、限制、不能违反的规则

## 具体信息
不写出来就会丢失的值：路径、参数、链接、数据、名称、工具版本、环境依赖等。多个产出文件时说明它们之间的关系

要求：条目式输出，每条自包含。新会话的 AI 看了就能直接接上。
```

## 设计说明

### 六段结构的来源

基于 Hermes Agent 上下文压缩模板（12段实战验证结构）精简而来：

| Hermes 压缩模板（12段） | 迁移 Prompt（6段） | 说明 |
|---|---|---|
| Goal | **目标** | 直接对应 |
| Constraints & Preferences | **关键约束** | 合并 |
| Key Decisions | **已确认的结论** | 直接对应 |
| Resolved Questions | 已确认的结论（吸收） | 合并 |
| Completed Actions | **进度 ✅** | 直接对应 |
| Active State | 具体信息（吸收） | 合并 |
| In Progress | **进度 🔧** | 直接对应 |
| Blocked | 已否决/具体信息（吸收） | 合并 |
| Pending User Asks | **进度 ⏳** | 直接对应 |
| Relevant Files | **具体信息** | 合并 |
| Remaining Work | **进度 ⏳** | 直接对应 |
| Critical Context | **具体信息** | 直接对应 |

### 四状态阶段判断

来自 LifeOS 运行时框架。让 AI 先判断用户在做什么类型的事：
- **发散探索** — 脑暴、探索可能性、收集信息。侧重保留方向性判断
- **收敛执行** — 确定方案、写文章、改代码。侧重保留操作进度
- **校验纠错** — 检查结果、改 bug、调整方向。侧重保留错误上下文和修复思路

### 测试结果

用3个真实长会话做对比测试，5维度评估（覆盖度、精准度、踩坑防护、可接续性、篇幅）：

| 场景 | V1 | V3 | 变化 |
|------|-----|-----|------|
| 技术操作（飞书wiki同步） | 9.4 | 9.2 | 持平 |
| 写作+研究+多任务混合（228条） | 3.2 | 4.55 | **+1.35** |
| 写作+配图+源码分析混合 | 4.4 | ~4.6 | +0.2 |

V3 解决了 V1 的三个核心缺陷：

1. **多任务并行支持** — "如果有多条任务线，分别标注"
2. **写作场景的软否决** — "迭代中不认可的角度/写法/方向"
3. **多工件状态追踪** — "涉及多个文件时，列出每个文件各自状态"

### 迁移时机建议

- 要求高质量的任务（写文章、系统设计）：上下文到 40-50% 时迁移
- 普通对话/问答：50-60% 可以
- 探索性对话：60-70%
- Agent 用户（工具调用多）：比 Chatbot 用户提前 10-15%
